# 42586 기능개발

## 문제 설명

- 기능은 진도가 100% 진행된 다음에 서비스에 반영할 수 있음.
- 각 기능의 개발속도는 다르기 때문에, 뒷 기능이 먼저 개발될 수 있지만 배포는 앞 기능이 배포될 때 가능
- 배포되어야 하는 순서대로 작업의 진도가 적힌 정수 배열 progresses
- 각 작업의 개발 속도가 적힌 정수 배열 speeds
- 배포마다 몇 개의 기능이 배포되는지 return 하는 solution 함수 구현

## 제한 조건
- 작업의 개수(progresses, speeds배열의 길이)는 100개 이하입니다.
- 작업 진도는 100 미만의 자연수입니다.
- 작업 속도는 100 이하의 자연수입니다.
- 배포는 하루에 한 번만 할 수 있으며, 하루의 끝에 이루어진다고 가정합니다. 
  - 예를 들어 진도율이 95%인 작업의 개발 속도가 하루에 4%라면 배포는 2일 뒤에 이루어집니다.

## 입출력 예시
1. 예시
   - progresses	: [93, 30, 55]
   - speeds	: [1, 30, 5]
   - return : [2, 1]
     - 1번 기능 93, 2번 기능 30, 3번 기능 55가 현재 진행도
     - 작업속도를 보면, 1번 기능은 7일, 2번 기능은 3일, 3번 기능은 9일뒤 배포 가능
     - 따라서 1번기능과 함께 2번기능이 배포 / 3번 기능은 9일뒤 따로 배포

2. 예시
   - progresses	: [95, 90, 99, 99, 80, 99]
   - speeds	: [1, 1, 1, 1, 1, 1]
   - return : [1, 3, 2]
     - 1일 후 3, 4, 6 기능 배포가능
     - 5일 후 1 기능 배포가능 -> 혼자 배포
     - 10일 후 2 기능 배포가능 -> 2, 3, 4 배포
     - 20일 후 5 기능 배포가능 -> 5, 6 배포

### 문제 풀이
`1. 설계`
- 각 기능별 100가 되는 날짜를 먼저 계산하자
- 그리고 순서대로 배포 가능한대로 계산을 수행하면 될 것이라고 생각함

`2. 수정`
- 기준 배포날짜를 바꾸면 안됨.
- 1일차보다 작은애들을 전부 합쳐서 계산해야지, 기존날짜를 바꾸고 다음날짜보다 작은걸 쓰면 안된다는거임
- 문법적으로 더 깔끔한게 있을거라고 생각함
